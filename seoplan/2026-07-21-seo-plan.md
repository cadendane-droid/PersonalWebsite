## Ground rule this cycle operated under

No visible text was changed anywhere on the site, and no visual layout, look, or structure was altered. Every change below is either metadata that isn't rendered on the page (meta tags, structured data, robots.txt, sitemap.xml), a non-visual HTML attribute (loading priority), an invisible semantic tag swap (h1→h2, verified pixel-identical via inline styles), or an asset re-encode that preserves the original pixel dimensions and visual quality. Every view was screenshotted before/after with a headless browser to confirm zero visual diff.

## 3. SEO audit (from reading `index.html` and repo structure)

Findings, in rough order of impact:

1. **Zero discovery/social metadata.** No canonical URL, no `og:url`/`og:image`/`og:site_name`, no Twitter Card tags, no structured data (JSON-LD). Meta description and basic OG title/description existed and were reasonable, but that's it. Link previews (iMessage, Slack, LinkedIn shares) had no image and search engines had nothing but a title+description to build a snippet from.
2. **No `robots.txt` or `sitemap.xml`.** Nothing explicitly invites crawling or tells a crawler where the content is. Not necessarily blocking indexing (crawlers can still find the page), but it's a missing, free, standard signal.
3. **Massive, uncompressed images with a mis-prioritized hero image.** `assets/` was ~11.6MB. The hero portrait (`portrait.png`, the largest above-the-fold element — the Largest Contentful Paint candidate) was 1.5MB **and marked `loading="lazy"`**, meaning the single most important image for the page's Core Web Vitals score was actively deprioritized by the browser. Core Web Vitals (LCP in particular) are a confirmed Google ranking factor, and matter even more on the 68% of visits that are mobile.
4. **Duplicate `<h1>` tags.** Because all five "pages" (home/hobbies/pro/growth/projects) live in one `index.html` and are toggled with CSS rather than routed to separate documents, the DOM contained five `<h1>` elements simultaneously. That dilutes the page's topical signal to crawlers.
5. **No favicon.** Browser tabs and bookmarks show a blank/default icon. This is a real, user-visible change if fixed (it shows up in the browser chrome), so per this cycle's "no visible difference" constraint it was **not** touched — flagged below for an explicit go/no-go from Caden next cycle.
6. **Content is entirely single-URL.** The site is a client-side-toggled single-page app addressed by `#hash` fragments (`#hobbies`, `#pro`, etc.), not real distinct paths. Search engines generally collapse `#fragment` URLs into the root URL, so even though all the content is present in the initial static HTML (good — it's not JS-injected at the top level), Google can realistically only ever index and rank **one** URL for this whole site. That caps how much organic traffic is achievable without either (a) real per-section URLs/pages, or (b) accepting that everything must rank under the one homepage URL. Flagged for a larger future cycle — it's a structural/architecture question, not a safe drop-in fix.
7. **A meaningful share of the page's real text (timeline "moments", work experience, education, skills, project write-ups, media log, build notes) is rendered into the DOM by JavaScript at load time** (`el("experience").innerHTML = ...`, etc.) rather than present in the raw HTML. Googlebot generally executes JavaScript and can index this, but it's a slower, less reliable second rendering pass, and other crawlers/bots (many link-preview bots, some AI answer engines) don't execute JS at all and will only see empty containers. Converting this to server-rendered-at-build-time static HTML would be a genuine SEO upgrade, but it touches the content-rendering pipeline directly and carries real risk of visual regression if rushed — deliberately deferred to a future cycle with proper before/after visual diffing, not attempted in this pass.

## 5. Implementation report

All changes are on `index.html` (head + two attributes), two new root files, and in-place image re-encodes. Nothing in the visible page text, layout, colors, or structure changed — verified with Playwright screenshots of all five views before/after.

| Change | File(s) | Why |
|---|---|---|
| Added `<link rel="canonical">` | `index.html` | Tells search engines the one authoritative URL, avoids duplicate-content ambiguity. |
| Completed Open Graph tags (`og:url`, `og:site_name`, `og:locale`, `og:image`) | `index.html` | Link previews on LinkedIn/iMessage/Slack now show an image and correct metadata — directly aimed at turning shares into clicks. |
| Added Twitter Card tags | `index.html` | Same reasoning, for X/Twitter shares. |
| Added JSON-LD `Person` structured data | `index.html` | Helps Google understand "Caden Ice" as an entity (name, job, employer, alma mater, LinkedIn, location) — all facts already stated elsewhere on the page, nothing new asserted. Improves odds of a rich result / knowledge panel for name searches. |
| Added `theme-color` meta tag | `index.html` | Set to `#f4efe8`, the site's existing background color — matches what's already visible, no new color introduced. |
| Added `robots.txt` + `sitemap.xml` | repo root | Standard crawl signals; sitemap currently lists the one canonical URL (see audit point 6 — hash-fragment "pages" aren't real indexable URLs, so listing them would be incorrect). |
| Removed `loading="lazy"` from hero portrait, added `fetchpriority="high"` | `index.html` | The hero image is the LCP element and is above the fold on every load — it should never have been lazy-loaded. This is a same-image, same-position, same-size change; only load timing/priority changes. |
| Demoted the 4 non-home `<h1>` → `<h2>` (hobbies/professional/growth/projects headers) | `index.html` | One `<h1>` per document is standard practice; all heading font sizes are inline-styled so this is visually identical — verified in screenshots. |
| Re-encoded all JPEG/JPG photos in place (quality 84, progressive, same pixel dimensions) | `assets/*.jpg`, `*.jpeg`, `*.JPG` | ~32% smaller on average, visually indistinguishable at this quality level for photographic content. |
| Converted `portrait.png` → `portrait.jpg` (quality 88), updated the one reference | `assets/portrait.jpg`, `index.html` | The hero/LCP image: **1521KB → 105KB (93% smaller)**. This is the single biggest performance win in this cycle — directly improves LCP, which is a ranking factor and matters most on the 68% of traffic that's mobile. |

**Total `assets/` footprint: ~11.6MB → 7.3MB.**

**Explicitly not changed / deferred (queued for future cycles):**
- **Favicon** — would be visible in the browser tab, which this cycle's "no visible difference" rule blocks. Needs an explicit yes from Caden before a future cycle adds one.
- **Per-section real URLs / server-rendered routing** — the biggest structural lever for organic growth (would let Google index and rank `/hobbies`, `/projects`, etc. separately instead of collapsing everything into one URL), but it's an architecture change with real regression risk and arguably touches "structure," so it needs a deliberate, well-tested future cycle — not bundled into a routine automated pass.
- **Pre-rendering the JS-injected content sections into static HTML** — genuine SEO upside (see audit point 7), same reasoning: worth doing carefully with full visual diffing, not rushed.
- **PNG screenshots** (`resume.png`, `almura1-3.png`, `vault.png`) were left alone this cycle — they're below-the-fold, text/UI-containing screenshots where lossy re-encoding carries more legibility risk than the photographic images touched above. Good target for a future cycle with proper tooling (e.g. lossless WebP) and a visual check.
- **Alt text wording** — left untouched. Some alt text is generic (e.g. `alt="Travel"`) and could be more descriptive for image search, but writing new descriptions risks misdescribing photos I can't verify, and borders on "content," so it's out of scope for an automated pass.

## 6. Expected outcome / how to judge this cycle next time

These changes target **crawlability, indexability, and Core Web Vitals** — the prerequisites for organic traffic to exist at all — not content or backlinks, so the realistic expectation is:

- **Immediate (this week):** no meaningful traffic change expected; PostHog visitor counts are driven by direct/referral traffic which this cycle didn't touch.

Link shares (LinkedIn, text messages) should start showing a proper preview card with the portrait image instead of a bare title.
- **Ongoing:** faster load, especially on mobile (68% of current traffic), should show up as a small bounce-rate improvement in PostHog's web overview, independent of any traffic-source change.
